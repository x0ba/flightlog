FlightLog planning artifact / draft 1

Model and framework agnostic tracing

Decision baseline

FlightLog becomes a logging SDK and API first. Browser agents remain first-class, but tool-call agents, model calls, approvals, artifacts, and evaluations move into a versioned trace schema.

Goal

Expand FlightLog from a browser-agent flight recorder into a framework-neutral agent trace system. The product should let OpenAI, Anthropic, LangChain, Vercel AI SDK, and custom loops record the same operational story: goal, planning, model calls, tool call lifecycles, browser actions, observations, approvals, artifacts, results, and evaluations.

schema: flightlog.trace.v1 api: /api/v1 sdk: trace-first raw provider payloads optional

Non-goals

  1. Do not turn the app into a hosted agent runtime.
  2. Do not require a specific model provider or orchestration framework.
  3. Do not remove browser replay. Generalize around it.
  4. Do not preserve the current API shape at all costs. This migration may break it.

Target architecture

Agent app OpenAI, Anthropic, LangChain, Vercel AI SDK, or custom loop.
FlightLog SDK Small adapters normalize calls into trace events and spans.
Versioned API /api/v1/traces, append-only event ingestion, batch support.
Trace store Runs, spans, events, artifacts, evaluations, provider payload refs.
Trace viewer Timeline plus replay, tool lifecycle, model calls, approvals, evals.

Data model recommendation

Normalize the public model enough to query across frameworks, but keep raw provider payloads as optional debug data. The normalized record is the contract; raw payloads are best-effort evidence.

Recommendation: store provider, framework, model, and raw metadata on relevant spans/events, but build UI and evaluation logic from normalized fields first.

Core entities

  1. Trace: current run concept, with versioned schema metadata.
  2. Span: timed operation such as model call, tool call, browser step.
  3. Event: append-only point-in-time fact inside a trace or span.
  4. Artifact: screenshot, DOM snapshot, JSON, logs, prompt, output.
  5. Evaluation: rule and LLM findings over traces and spans.

Generic tool lifecycle

Browser actions remain first-class, but tool calls get a framework-agnostic lifecycle that can represent OpenAI function calls, Anthropic tool_use blocks, LangChain tool invocations, and Vercel AI SDK tool execution.

Phase 1: Versioned ingestion

  1. Add schemaVersion to traces/runs and API responses.
  2. Create /api/v1/traces routes alongside current run routes.
  3. Add server validation for flightlog.trace.v1.
  4. Introduce span storage and event names without deleting current UI paths.

Phase 2: SDK redesign

  1. Rename the primary object from Run to Trace.
  2. Add startSpan, endSpan, and lifecycle helpers.
  3. Add adapter entrypoints for OpenAI, Anthropic, LangChain, and Vercel AI SDK.
  4. Keep convenience helpers for browser screenshots and DOM snapshots.

Phase 3: Migration

  1. Write a one-way migration from runs/events to traces/spans/events.
  2. Map current browser_action events to browser spans plus lifecycle events.
  3. Map tool_call and tool_result into requested/completed pairs.
  4. Document breaking API and SDK changes in a migration guide.

Phase 4: Trace viewer

  1. Replace run detail internals with trace/span queries.
  2. Add event filters for model, tool, browser, approval, artifact, evaluation.
  3. Keep replay prominent for browser spans and show structured previews for other spans.
  4. Add side-by-side raw and normalized payload views in the inspector.

Adapter plan

  1. OpenAI: wrap Responses/Agents calls, capture model input/output, function calls, computer calls, usage, response id, and safety checks.
  2. Anthropic: wrap Messages calls, capture tool_use/tool_result blocks, stop reason, usage, model, and message ids.
  3. LangChain: provide callback handler support for LLM start/end, tool start/end, chain start/end, retriever events, and errors.
  4. Vercel AI SDK: provide middleware/callback helpers for toolCall, toolResult, stream chunks, usage, and final result.

Open questions

  1. Should traces support distributed child traces across services?
  2. Should raw provider payloads be stored inline or always as artifacts?
  3. Should the SDK expose OpenTelemetry export/import later?
  4. Which adapter should ship first after the core schema?

Evaluation changes

  1. Evaluate goal completion at trace level.
  2. Evaluate constraint violations from normalized event categories and artifacts.
  3. Detect repetition across tool and browser span signatures.
  4. Track human approval from approval.requested and approval.resolved.
  5. Add tool reliability metrics: failure rate, retries, timeout count, latency.

Risks

The main risk is over-normalizing too early. Keep the v1 schema small, preserve raw payload references, and let adapters mature from real examples instead of theoretical coverage.

  1. Breaking API migration can slow current UI work.
  2. Provider payloads may include sensitive data; redaction must be part of ingestion.
  3. Nested spans add complexity to simple SDK examples.
  4. Streaming model calls need careful partial-event handling.

Acceptance criteria for the first implementation pass

  1. A new trace can be created through /api/v1/traces with schemaVersion.
  2. A tool call can be logged with requested, started, completed, and failed lifecycle events.
  3. A browser action can still be displayed as a first-class replay event.
  4. The dashboard can show a mixed trace containing model calls, tool calls, browser actions, approvals, and artifacts.
  5. The SDK can log one provider-neutral trace without importing OpenAI, Anthropic, LangChain, or Vercel AI SDK packages.
  6. At least one adapter proof of concept maps provider/framework data into the normalized schema.
  7. Evaluation still reports goal completion, constraint violation, repetition, and human approval.
